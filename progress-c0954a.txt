ARCHITECTURE (done)
===================
  - NavScope.java: per-rule scratch pad; allocates real SQL aliases eagerly via ctx.aliases()
    at PropertyAccessLowering discovery time (option B from plan §2.4). Dedups by path prefix;
    parents tracked via Entry.parentPrefix for chained lookups.
  - LoweringContext surface (6 fields, 11 methods):
      mappings, mode, variableBindings, env, aliases, navScope
      bindVar, withNavScope, lookupVar, lookupBinding, aliases(), navScope(),
      storeFor, nextAlias, mappings(), mode(), variableBindings()
  - PropertyAccessLowering: single path. Reads VarBinding.store() for root, walks
    associationPath resolving non-embedded hops via NavScope.navigate(prefix, parentPrefix, jr).
    Throws PlanGenNotPortedException (never falls back) on: non-variable source, unbound
    variable, non-Identifier binding, unresolved association, or non-embedded hop without
    NavScope (join conditions per plan §2.5).
  - Relations.install(source, srcAlias, srcStore, scope, ctx): single overload.
    Honors per-nav parent alias via scope.lookup(entry.parentPrefix()). Binds join-cond
    sourceParam to parent alias (not always srcAlias — fixes latent chained-prefix bug).
    Binds sourceParam to parentStore (srcStore for root), targetParam to j.targetResolution().
  - FilterLowering / ProjectLowering / SortLimitLowering / GroupByAggregateLowering
    (GroupBy + Aggregate) / ExtendLowering: all migrated to bindVar + shared NavScope +
    Relations.install(scope).
  - JoinLowering: per-side bindVar with each side's store. Fixes plan §2.1 latent bug
    where both left and right params bound against left's store. Accounts for the +1 test.
  - ExtendLowering: traversalHops pre-register into NavScope as synthetic [__hop_i] entries
    with empty parentPrefix (matches legacy flat-install semantics: each traverse hop's
    source binds to srcAlias). aliasChain built from NavScope.navigate return values so
    multi-param extend lambdas ({src, hop1, hop2 | ...}) still positional-bind correctly.
    One Relations.install call weaves both traversalHops + discovered-assoc navs.
  - ControlFlowLowering.let: bindVar(name, value, null) — let values aren't row sources.

REMAINING CLEANUP (audited — all three are load-bearing, not dead code)
========================================================================
  - StructLowering struct-extract-on-assoc rewrite branch: AUDITED by replacing with a
    throw. Fired ~325 times across the engine suite — the compiler emits
    TypedStructExtract(TypedPropertyAccess) for association-struct access rather than a
    flat TypedPropertyAccess with full path. Branch KEPT. The cleaner upstream fix is
    to have the compiler emit a flat TypedPropertyAccess(path=[firm, legalName])
    directly; until then the rewrite is mandatory.
  - TypedExtend.traversalHops: AUDITED. Misread in plan §6 — these are top-level
    traverse() clauses in extend parameters (syntactic construct distinct from
    association navigation inside scalar extend bodies). Multi-param extend lambdas
    ({src, hop1, hop2 | expr}) bind positionally to srcAlias + each hop's allocated
    alias. NavScope covers assoc nav inside scalar bodies; traversalHops covers the
    extend-level traverse() syntax. Both are needed. Field KEPT.
  - TypedAssociationExtendCol / TypedEmbeddedExtendCol marker cols: AUDITED. Consumed
    by MappingResolver (builds JoinResolution from TypedAssociationExtendCol.hops and
    embedded subs from TypedEmbeddedExtendCol.subColumns). PlanGen correctly returns
    null — the physical JOIN is installed by MR into the mapping's sourceRelation
    extends, which PlanGen lowers through the normal path. KEPT.
  - Stale doc comment in ExtendLowering.lowerExtendCol fixed (said "installed by
    traversalHops" — actually installed by MappingResolver into sourceRelation).
